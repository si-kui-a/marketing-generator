# 協作模式觀察記錄

> 本檔案的內容全數可追溯至marketing-generator專案對話串中的具體訊息,
> 不包含臆測。每一條觀察都附上發生的情境作為依據。

---

## 1. 溝通與指令模式

### 1.1 高度重複使用固定範本指令,且會逐字複製貼上

**觀察**:從Phase2開始,幾乎每一輪的訊息都以完全相同的文字開頭:「直接產出指令包
同時找出Wöchentlich‧alle Modelle較為耗費的環節...」,逐字重複超過六次。

這是優點也是風險:

- **優點**:指令意圖高度穩定,不需要每次重新確認「這次要不要也做治理總結」,可以直接視為常駐要求執行
- **風險**:完全相同的文字重複貼上,曾經在Phase4被執行端(Claude Code)誤判為「疑似殘留/重複貼上的雜訊」而暫停處理其中一部分內容(governance skill v1.2草稿的部分),因為執行端在自己的session記錄裡找不到對應的新事件依據。這不是使用者的錯,而是逐字重複的範本指令,在缺乏額外上下文標記時,執行端難以區分「這是新一輪的正式要求」還是「不小心貼重複了」

**建議因應方式**:若這個範本指令要素持續每輪出現,未來協作中可考慮:每次除了範本文字外,額外附加一行當輪的具體差異(如本輪新增的第三項要求),讓執行端更容易錨定「這輪新增了什麼」,降低被誤判為雜訊殘留的機率。

### 1.2 對多選項問題傾向給出極簡短的確認回覆

**觀察**:本對話中大量出現單一詞彙或字母作為回覆,如「是」「同意」「確認」「A」「ok」,用來回應先前提出的結構化多選項問題(如GitHub優先架構的四點確認、local_kit五模組的三項決策等)。

這代表:使用者信任把選項拆解清楚後,直接做決策不需要展開論述。這種模式運作良好的前提是問題必須拆成「選A或選B」這種可以被單字回答且不產生歧義的結構——本對話中採用的表格對照、編號選項、單一問句限制(一次最多問一件事)的做法,與這個回覆習慣是匹配的,應該延續。

**風險點**:若某次把問題問得不夠精確(例如同時問了兩件不相關的事情,卻只用一個問句包裝),極簡回覆可能無法明確對應到原本想問的兩個子問題,容易產生誤解。這件事本身沒有在本對話中造成實際錯誤(因為問句設計還算嚴謹),但這是一個需要持續注意維持的協作前提,非使用者的責任。

### 1.3 傾向直接採納優先順序建議,不逐項反駁

**觀察**:當規劃端在「推薦現在最應執行步驟」時提供了一個帶理由的比較表(Phase6/7/8.1的依賴狀態與風險評估),使用者直接回覆「直接產出指令包」,未逐項質疑或要求調整推薦邏輯。

這代表:使用者傾向把「決定優先順序」這件事的認知負擔外包給規劃端,前提是取捨的理由要講清楚(表格化呈現依賴關係、風險、立即價值)。這個模式運作良好,代表這種呈現方式(比較表優於純文字論述)對這位使用者特別有效。

## 2. 決策與治理習慣

### 2.1 對「安全機制是否為技術強制」的堅持

**觀察**:從local-kit架構討論階段開始,使用者連續多輪主動質疑「這條規則是不是只是寫在文件裡,AI是否真的會遵守」,直到每一條規則都轉譯成可測試、可被git hook強制擋下的程式碼機制,才停止進一步質疑。

這是本對話中最一致、最貫穿全程的一條決策原則:**不接受「口頭約定」作為安全保證,只接受「程式碼層級的強制」**。這個標準應該延續套用到所有未來涉及安全性/資料完整性的討論,不需要每次重新驗證這個偏好是否仍然成立——它在本對話中從Day1到Phase8.1從未動搖過。

### 2.2 主動要求「誠實揭露侷限」而非追求表面完美

**觀察**:多次在指令包遇到技術限制時(如Python `http.server`單執行緒無法做真正的伺服器端逾時中斷),使用者從未要求「想辦法解決」,而是接受「在文件中誠實註明限制」作為合理的收尾方式。

這代表:使用者對「功能是否完美」的容忍度,遠高於對「文件是否誠實反映實際限制」的容忍度。未來遇到類似的架構限制,優先選擇「誠實文件化限制」而非「為了看起來完整而過度工程化去繞過限制」,會更符合這位使用者的偏好。

### 2.3 願意讓執行端(Claude Code)質疑並修正規劃端的錯誤

**觀察**:本對話中至少發生兩次,執行端在實際執行指令包時,發現規劃內容本身有架構性錯誤(Phase1 §E的資料誤刪事故、Phase4的v1.2草稿可信度質疑),使用者在事後稽核時,明確肯定執行端的質疑行為是正確的,而非因為「規劃端說的」就要求執行端照做。

這代表:協作結構預期執行端有能力且應該對規劃內容本身保持批判性,不是單純的上對下執行關係。這個期望應該持續傳達給任何未來負責執行的AI實例——不應該假設規劃內容一定正確,發現矛盾時優先提出而非默默照做或默默修正。

## 3. 容易產生歧義或風險的具體行為模式(供警示,非批評)

### 3.1 操作環境(PowerShell/`.env`)存在具體的、可預期的操作誤區

**觀察**:本對話前段記錄了以下具體誤操作,且部分重複發生:

- 誤在錯誤的PowerShell視窗(未`cd`到專案目錄)執行指令
- 誤將真實金鑰貼入`.env.example`而非`.env`(發生兩次)
- 誤以檔案總管雙擊或`localhost:8765/index.html`路徑開啟前端頁面(server不提供靜態檔案路由)
- 誤將另一個不相關專案的`.env`內容當成marketing-generator的設定

這不代表能力不足,而是這類「多視窗、多檔案、多專案」的操作情境天生容易混淆,任何人在缺乏視覺化引導的純文字指令下都可能發生類似狀況。

**具體警示(供未來協作直接套用)**:

1. 每次要求在PowerShell執行指令前,明確要求先確認視窗標題列或`pwd`輸出,不能假設已經在正確目錄
2. 涉及`.env`編輯的指示,應該明確標註「這是`.env`不是`.env.example`」,並建議用`Get-Content`確認檔名後才開始編輯,而非只給檔案路徑
3. 涉及前端頁面開啟方式,需要同時給出「用瀏覽器直接開啟本機檔案」與「透過server路由訪問」兩種情境的區別說明,不能只給一種指示後假設對方理解背後原理

### 3.2 對「已完成的懸案」的追蹤容易被新一輪要求覆蓋、需要主動複核

**觀察**:`start.bat`雙擊測試、`bindHoldToDelete()`3秒計時測試這兩項懸案,在對話中段被口頭回報「成功」「正常可用」,但這個回報只存在於規劃對話(本對話串),執行端(Claude Code)在後續某一輪的獨立session中對這兩項的追蹤記錄依然顯示「未驗證」,直到規劃端主動介入釐清才化解矛盾。

這不是使用者的錯(規劃端才有責任確保資訊正確傳遞給執行端),但這反映一個值得注意的協作模式:傾向在對話中口頭確認完成事項,而不會主動要求「這個確認要不要也寫進正式的追蹤文件」。未來協作中,規劃端應該在口頭回報測試結果的當下,就主動詢問或直接記錄「這個確認需要一併同步到哪些文件/哪個執行實例」,而非被動等待矛盾出現才處理。

## 4. 使用本檔案的方式

與此使用者展開marketing-generator(或架構/協作模式相近的專案)新一輪協作前,應先讀本檔案第1、2節校準溝通與決策風格的預期,並對照第3節的具體警示,主動在給出操作指示或懸案追蹤時採取對應的預防措施,而非等問題重演才處理。
